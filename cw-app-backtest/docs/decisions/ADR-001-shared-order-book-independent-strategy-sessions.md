# ADR-001：共享订单簿与独立策略执行会话

## 状态

已接受，2026-07-23。

## 背景

原回测链路按股票只注册一个 `OrderBookSession`，其中同时保存盘口、静态历史事实、交易模板和一个有符号的 `transactionStatus`。这种结构能运行单一模式，但存在三个限制：

- 同一股票无法让多个模式同时判断，后注册模式会覆盖前一个模式的状态。
- 盘口热数据与策略持仓、买卖状态绑定，无法安全复用于低频策略。
- 模板初始化时按几进几和市值选择卖出类，导致板位场景持续扩张。

本轮范围只包含回测及共享核心模块；实盘交易继续使用兼容入口，后续单独迁移。

## 决策

采用以下边界：

1. `OrderBookSession` 只代表一只股票、一个交易日的共享市场会话，持有 `MarketSessionSpec` 与唯一的热 `OrderBook`。
2. 同一市场会话可以注册多个 `StrategyExecutionSession`。每个策略会话分别持有 `StrategySessionKey`、`TradeStaticFacts`、`TradeExecutionTemplate` 和 `TradeExecutionState`。
3. `TradeExecutionState` 内部分离买入状态、卖出状态和持仓状态；原整数状态只作为旧调用链兼容映射。
4. 沪深 Handler 每条行情只更新一次共享盘口，然后按注册顺序把事件分发给所有订阅该触发类型的策略会话。
5. `UnifiedTradeExecutor` 统一执行连续竞价买入、连续竞价卖出、均价卖出和状态推进，并输出带 `strategySessionId`、`strategyId` 的 `TradeOrderIntent`。
6. 模板携带昨日板高、目标板高、市值档、历史 K 线、前一板加速状态、集合竞价许可和最早盘中买入时间。卖出从统一规则目录读取这些事实，不再在模板编译时选择某个几进几执行器。
7. 模板声明 `TradeTriggerType`。现有三模式订阅竞价、逐笔、分钟和尾盘触发；未来低频模式可只订阅日线或定时触发，避免进入逐笔热路径。

## 固定事件顺序

每条行情固定按照以下顺序处理：

1. 更新共享订单簿。
2. 按注册顺序执行各策略模板。
3. 生成带策略来源的订单意图。
4. 由回测成交网关记录或撮合。
5. 只更新对应策略会话的买入、卖出和持仓状态。

订单簿不能保存策略资金、持仓或买卖状态；账户资金竞争与仓位仲裁属于回测账户或未来实盘执行网关。

## 兼容与迁移

- 原 `OrderBookSession(spec, facts, orderBook, template, state)` 构造器保留，内部创建一个主策略会话。
- `transactionStatus()` 与 `transactionStatus(int)` 保留，映射到分离后的状态对象。
- `OrderExecutionGateway` 原动作方法保留；新增的 `execute(TradeOrderIntent)` 默认适配旧实现。
- 已验证的几进几规则实现暂时作为统一规则目录的规则提供者保留，避免在实盘尚未迁移时破坏兼容入口；回测模板不再持有预选中的板位策略实例。

## 后果

优点：盘口重建唯一、多策略状态隔离、订单与规则可归属、低频策略可跳过逐笔、以后回测和实盘能复用同一核心状态模型。

代价：一个行情事件需要向多个会话分发；多策略共用真实账户时还需要独立的资金和优先级仲裁；迁移卖出规则时必须继续使用固定候选逐笔比对，不能只比较汇总收益。

## 验收

- 核心测试证明同一共享盘口可同时驱动两个独立策略会话，二者分别推进卖出状态并产生不同会话标识的订单意图。
- 三个模式继续使用固定选股候选执行全量回测。
- 卖出结果按 `positionType|symbol|tradeDate` 比较卖出时间、价格、规则、收益和状态。
